# GCPS Theme Studio

A self-contained **R Shiny** app that builds a **complete GCPS theme** from any
source color — one of the seven analytics **bases**, a high-school **cluster**,
or an individual **school** (which inherits its cluster color). Layer
**typography**, **surfaces**, and **spacing** onto a **palette**, preview the
whole theme on a live K-12 dashboard, check **WCAG contrast**, and export it for
**Power BI**, **R / ggplot2**, and **CSS** in one pass.

The tool runs across six tabs: **Palette**, **Typography**, **Surfaces**,
**Theme Preview**, **Accessibility**, and **Export**.

Palette types generated (deterministically, in OKLCH):

- **Sequential** — single-hue ramp, light → dark (5 stops)
- **Tints & Shades** — 11-step UI scale (50–950)
- **Diverging** — two-ended through a neutral center
- **Categorical** — distinct hues from the source via **color-theory harmonies**
  (Analogous, Complementary, Split-Comp, Triadic, Tetradic, Qualitative), plus
  the curated **GCPS Curated** and **All Clusters** sets, and fixed-identity
  **Race/Ethnicity** and **School Level** palettes
- **Performance** — ordinal proficiency scale, **4 / 5 / 6** levels, in a fixed
  **Semantic** (good→needs-support) or **Source-tinted** monochrome variant, or
  a **Milestones gradient** (Warm-to-Cool, **3 / 5 / 7** ordered stops, no
  achievement names)
- **Continuous** — fine 9-step gradient
- **Trend** — fixed positive / negative / neutral indicators

## Run locally

```r
# from this directory
install.packages(c("shiny", "htmltools", "jsonlite"))
shiny::runApp(".")
```

…or open `app.R` in RStudio and click **Run App**. No build step, no Node, no
bundler.

**Rendering:** the tool is rendered client-side by a small JS engine in `www/`
(`theme-studio.js` + `theme-studio-app.js`, backed by `palette-data.js`).
`app.R` is the Shiny entrypoint that serves the page (keeping it a valid Shiny
app for Posit Connect) and sources the R palette engine, which mirrors the JS
and stays available for server-side generation/export.

**Fonts:** the UI uses Google Fonts (Spectral, Source Sans 3, Lexend, IBM Plex
Sans/Mono) via a `<link>` in `app.R`. The CSS stacks fall back to system
families, so the app still renders if the fonts are blocked (e.g. on a
network-locked Posit Connect instance) — it just won't look identical.

## Project structure

```
gcps-theme-studio/
├── app.R                      # Shiny entrypoint: serves the page + minimal server
├── R/
│   ├── gcps_palette_data.R    # GCPS_BASE, CLUSTERS, SCHOOL_CLUSTER, HARMONY, …
│   └── gcps_palette_library.R # OKLCH generators + gcps_build_palette() + exporters (mirrors the JS)
├── www/
│   ├── theme-studio.css       # studio styles (the example design's stylesheet)
│   ├── palette-data.js        # JS mirror of R/gcps_palette_data.R (GCPS bases, clusters, schools)
│   ├── theme-studio.js        # engine: OKLCH palettes, WCAG, typography, surfaces (window.TS)
│   └── theme-studio-app.js    # 6-tab UI state, rendering, and event wiring
├── GCPS Theme Studio.html     # standalone reference build of the same UI
├── manifest.json              # Posit Connect manifest (regenerate — see below)
├── .gitignore
└── README.md
```

The reference UI is `GCPS Theme Studio.html` (standalone); the Shiny app in
`app.R` renders the identical markup and loads the same three `www/*.js` files.
The `R/` palette engine mirrors the JS engine (`theme-studio.js`) so a palette
built in the UI matches `gcps_build_palette()` in R.

## Deploy to Posit Connect

This is a **Shiny** app (it has server-side compute), so deploy it as a Shiny
content type — not as static content.

**A. Push-button (RStudio / Posit Workbench)** — open the project, click the
**Publish** button, choose the Connect server, publish the app.

**B. Programmatic (rsconnect)**

```r
# install.packages("rsconnect")
library(rsconnect)
# one-time: register the Connect server (connectServer / connectApiUser)
rsconnect::writeManifest(appDir = ".")          # regenerate manifest.json
rsconnect::deployApp(
  appDir        = ".",
  appPrimaryDoc = "app.R",
  appName       = "gcps-theme-studio",
  appTitle      = "GCPS Theme Studio"
)
```

**C. Git-backed** — commit `manifest.json` + all files and point Connect at the
repo. Keep `manifest.json` in sync with `rsconnect::writeManifest()` whenever
files change.

The committed `manifest.json` is a **starting point**; the authoritative copy is
regenerated by `rsconnect::writeManifest(appDir = ".")`, which records the exact
package set (ideally from a project `renv.lock`) that Connect restores.

### Reproducible packages (recommended)

```r
# install.packages("renv")
renv::init()        # snapshot shiny, htmltools, jsonlite (+ deps) into renv.lock
renv::snapshot()
rsconnect::writeManifest(appDir = ".")   # manifest then records the locked set
```

## No secrets, no data

The app ships **zero data and zero credentials** — palettes are generated in R
from the embedded GCPS color definitions. If a future version adds a live data
feature, read connection details from **Connect environment variables** via
`Sys.getenv()` (and use the `config` package keyed on `R_CONFIG_ACTIVE`); never
hard-code DSNs or passwords.

## Verifying the palette engine

`R/gcps_palette_library.R` ends with an **ACCEPTANCE** block of exact expected
hex values. To confirm the engine is correct on your machine:

```r
source("R/gcps_palette_data.R")
source("R/gcps_palette_library.R")
stopifnot(identical(
  gcps_sequential("#007C91")$hex,
  c("#CDE7ED", "#96C5D1", "#559FB1", "#007C91", "#005C70")
))
stopifnot(identical(gcps_source_color("schools", "Alcova ES"), "#CCCC99"))  # Dacula
```
